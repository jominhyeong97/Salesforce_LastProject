### 주요 기능 1 — 영업 콕피트 + 자연어 입력바 (AI 라우터)

**Page A — 무엇·왜**
> AE 김민수가 오늘 무엇을 해야 하는지 한눈에 보고, 한 줄 자연어로 묻고 한국어 답을 받는 통합 홈 화면.

도입 이유 — 영업의 일과가 메일·엑셀·메신저·ERP 4곳에 흩어져 있어 "지금 뭐가 급한지" 파악에 시간이 든다. 한눈 5카드(우선 처리 Lead / 위험 주문 / 견적 추적 / 미수금 액션 / 영업 파이프라인)로 카드화하고, AE가 자연어로 묻는 입력바 한 개를 둠.

`설계 철학 태그: AE 의사결정 중심 — 입력은 자연어 한 줄, 데이터 인용은 시스템이 책임`

**Page B — 핵심 로직**

자연어 입력 → 3-step AI 라우터:
1. 1단계 — 어느 데이터를 가져올지 결정 (Prompt Template: AgentRouter)
2. 2단계 — Apex 메서드 호출 (5종 Action 중 하나 자동 선택)
3. 3단계 — 결과를 한국어 자연체로 합성 (Prompt Template: AgentSynthesize)

코드 발췌 (5줄, 라우터 진입):
```apex
String tool = LlmRouter.choose(question);        // 1단계 — 도구 선택
Map<String,Object> data = invokeAction(tool, q); // 2단계 — 데이터 조회
return LlmRouter.synthesize(question, data);     // 3단계 — 자연어 응답
```

키워드 라우터(빠른 경로) + LLM 라우터(자유 질의) + Agentforce 응답(외부) 모두 동일 Apex 메서드 호출 — 진입점만 3개, 처리는 한 곳.

**Page C — 시연 영상**

"대성기공(주) 물건 나갔나요?" → 자동으로 고객사 요약 조회 → "미출고 주문 3건, 장기 미수금 보유" 응답 (15~30초 gif)

### 주요 기능 2 — 사업자등록증 OCR (Prompt Builder 멀티모달)

**Page A — 무엇·왜**
> Opp(영업 기회) 단계에서 사업자등록증 PDF/이미지를 업로드하면 사업자번호·대표자·법인명·주소 4필드가 Account에 자동 보강된다.

도입 이유 — 한국 B2B 거래는 사업자번호 확보가 필수지만 Lead 단계에서 요구하면 단순 견적 문의자가 빠진다. 의사가 확인된 Opp 단계에 옮겨두고, 입력 자체도 사진 한 장 업로드로 자동.

`설계 철학 태그: 한국식 정서 + AE 의사결정 중심 — 사업자번호 자동 추출, AE는 검토만`

**Page B — 핵심 로직**

```apex
wv.value = new Map<String,Object>{ 'Id' => contentDocId };
input.inputParams = new Map<String, ConnectApi.WrappedValue>{
    'Input:ContentDocument' => wv

ConnectApi.EinsteinLLM.generateMessagesForPromptTemplate(
    'BusinessLicenseExtract', '', input);  // 멀티모달 3-arg
```

응답 JSON 파싱 후 Account 업데이트. LLM 응답이 코드 펜스로 감싸 오는 경우를 대비해 stripCodeFences 헬퍼로 정제.

### 주요 기능 3 — AI 견적 작성 (Quote Builder + 의도 분류 + 자동 재견적)

**Page A — 무엇·왜**
> 견적 라인을 인라인 편집하고(할인은 금액 입력 시 % 자동 계산), 한국식 PDF로 발송하고, 고객 응답 메일의 의도("재견적"·"수락"·"기타")를 AI가 분류해 재견적이면 새 견적 초안을 자동으로 만들어둔다.

도입 이유 — 한국 영업의 견적은 ₩ 금액 단위가 우선이고 일금 한글 표기도 함께 들어간다. 재견적 응답이 오면 AE가 처음부터 다시 작성해야 했음 → AI가 의도 인지 + 가격은 그대로 복제 + 협상 메모만 자연어로 자동 메모.

`설계 철학 태그: 한국식 정서(₩·일금 한글) + AE 의사결정 중심(AI 분석만, 가격 결정은 AE)`

**Page B — 핵심 로직**

이메일 트리거 → 비동기 의도 분류 → 재견적이면 자동 초안:

```apex
// EmailMessage after insert
EmailIntentClassifyService.classifyAsync(em.Id); // 의도 분류 (Prompt)
// 결과: Intent__c = '재견적'
QuoteRequoteService.autoDraftIfRequote(em.Id);   // 새 견적 초안 생성
// 원본 라인 그대로 복제 + Description 에 AI 협상 분석 블록
```

가격 안전 정책 — AI는 견적 설명(Quote.Description)에 자연어 분석만 적고, 단가·수량·할인은 원본 라인을 그대로 복제. AE가 수동 확정.

**Page C — 시연 영상**

견적 발송 → 고객 응답 "5% 인하 부탁" → 의도 = '재견적' 배지 → 새 견적 자동 초안(`(재견적)` 접미사) → AE 검토 (30초 gif)

### 주요 기능 4 — L2C 자동화 (견적→주문→ERP 갭 보고)

**Page A — 무엇·왜**
> 견적 상태가 '수락'으로 바뀌면 견적 라인이 주문 라인으로 자동 복사되고, 수주 단계 진입과 동시에 주문이 자동 생성되며 약속 납기일이 주문에 그대로 승계된다. ERP 동기화 한 번이면 주문별 예상 출고일과 사유(미생산·미출고·재계획·정시 등)가 자동 표시된다.

도입 이유 — 입력 부담을 0으로 만드는 게 본 시스템의 핵심 가치. AE는 영업 기회 단계에서 약속 납기일을 한 번 입력하고, 그 후로는 승인·통보 같은 의사결정만.

`설계 철학 태그: AE 의사결정 중심 + 단계별 가이드("납기일을 저장해주세요" 게이트 메시지)`

**Page B — 핵심 로직**

견적 상태 변경 트리거 + 영업 기회 수주 트리거 + ERP 갭 풀:

```apex
// 견적 '수락' 전이 시 — 라인 자동 복사
if (q.Status == '수락' && oldStatus != '수락') {
    copyQuoteLinesToOpportunityProducts(q.Id);
}
// 영업 기회 수주 전이 시 — 주문 자동 생성 + 약속 납기일 승계
toCreate.add(new Order(
    Opportunity__c = opp.Id,
    EndDate        = opp.Promised_Delivery_Date__c, // 약속 납기일 → 주문
    ...
));
```

ERP 갭 — 약속 납기 대비 분기별 후보 풀(위험: D+3/+5/+7, 대형지연: 미생산·미출고·재계획)에서 주문 고유 ID 해시로 결정론 선택. 같은 주문은 매번 같은 결과.

**Page C — 시연 영상**

영업 기회에 납기일 6/15 입력 → 견적 작성 → 라인 자동 복사 → 수락 → 주문 자동 생성 → ERP 동기화 → "예상 출고 2026-06-20 (약속 6/15 → D+5 지연)" 표시 (40초 mp4)

## 4. 트러블슈팅 5건 (각 1페이지)

각 페이지 = 문제 / 원인 / 해결 / 학습. 내부 의사결정 번호는 빼고 일반 어휘로.D38 Opp.Promised_Delivery_Date__c → Order.EndDate 승계

### 1. LLM 응답 코드 펜스 문제

- 문제: Apex 단위 테스트는 33/33 통과, 실제 호출 시 JSON 파싱 실패 (` Unexpected character '`' code 96`)
- 원인: LLM이 응답을 ` ```json ... ``` ` 마크다운 코드 펜스로 감쌌음. "마크다운 절대 금지" 지시도 무시. 단위 테스트의 mock JSON은 펜스가 없어 못 잡음
- 해결: 모든 LLM 응답 wrapper에 stripCodeFences 헬퍼 통과 의무화
- 학습: **Apex 테스트 통과가 실제 LLM 통과를 보장하지 않는다**. mock 응답과 실제 응답이 다를 때를 항상 가정

### 2. 표준 주문 ↔ 영업 기회 연결 필드가 비활성

- 문제: 수주 시 주문 자동 생성 로직이 Apex 컴파일은 통과, 런타임에 "Invalid field" 예외
- 원인: 이 조직은 표준 주문→영업 기회 연결 필드가 활성화되어 있지 않음. Apex 컴파일러는 메타 캐시로 통과시키지만 런타임 schema는 다름
- 해결: 커스텀 lookup 필드(`Opportunity__c`)를 주문에 새로 만들고 트리거에서 이 필드를 채움
- 학습: 컴파일 통과 ≠ 런타임 통과. 표준 필드도 조직마다 활성 여부 다르니 Schema describe로 검증 후 결정

### 3. Agentforce 메타 직접 작성의 함정

- 문제: Agent 메타 XML 직접 조립 시 6건 연속 실패 (필수 필드 누락, enum 불일치 등)
- 원인: v66 Agentforce schema가 안정화 전, 문서 부족
- 해결: `sf agent generate agent-spec` + `sf agent create --spec` CLI 경로로 위임. SF가 일관된 메타 생성 → retrieve로 소스에 자산화
- 학습: 불안정한 schema는 CLI 도구에 위임. trade-off는 Agent 타입이 ExternalCopilot으로 제한된다는 점 — 운영 환경에서 Employee Agent로 전환 가능

### 4. 약속 납기일을 어디에 둘 것인가

- 문제: 처음엔 ERP가 납기일을 만드는 모델로 출발. 하지만 "영업 약속을 ERP가 만든다"가 도메인적으로 어색
- 원인: 책임 경계 미정립
- 해결: 도메인 인사이트 — **납기는 영업 약속(영업 기회의 속성)이고, ERP는 그 약속을 지킬 수 있는지 갭만 보고한다**. 약속 납기일은 영업 기회에 단일 필드로 두고, 견적·재견적·주문은 그 필드를 참조
- 학습: Single Source of Truth + 책임 경계 정립으로 회귀 범위와 기능 범위가 동시에 줄어듦

### 5. ERP 외부 통신 0 — 게이트웨이 패턴

- 문제: ERP·생산·품질을 직접 연동하면 시연 안정성도 떨어지고 평가 시 외부 변수도 늘어남
- 원인: 영업 시스템이 ERP 영역까지 책임질지 미정립
- 해결: ERP 인터페이스(`IExternalErpService`) + Mock 구현체 + Apex 내부 결정론. 실제 어댑터로 swap 가능
- 학습: Sales Cloud는 ERP 신호의 소비자다. 게이트웨이로 책임 격리하면 외부 callout 0으로도 통합 시연이 자연스러움

## 5. 아쉬웠던 점 (1페이지, 4건)

1. **Agentforce 외부 챗봇의 권한 한계** — Agent Type이 ExternalCopilot으로 강제되어 Apex Invocable을 외부에서 직접 호출하는 권한 메타가 일부 비공개. 관리자 경로로는 정상 동작하지만, 외부 챗봇 라이브 시연은 narrative로 우회. 운영 환경에서 Employee Agent 전환 시 해결 가능
2. **견적서 PDF 자체에 납기일 미인쇄** — 약속 납기일이 영업 기회와 주문에는 흐르지만 견적서 출력물에는 아직 안 들어감. 영업 관행상 견적서 자체에 납기일 명시가 자연스러우므로 후속 보완
3. **이메일 추적 픽셀 미배포** — 견적 발송 후 고객이 메일을 열었는지·링크를 눌렀는지 추적하는 백엔드는 완성되었지만, 외부에서 접속 가능한 Site 인프라 셋업이 미완. 백엔드 코드는 그대로 두고 Site만 활성화하면 됨
4. **자유 텍스트 납기 표현의 자동 날짜 변환** — Lead 단계에서 "다음달 말까지" 같은 자유 표현은 자동 추출되지만 정확한 날짜 변환은 미구현. AE가 영업 기회 단계에서 수동 입력. 후속 Prompt 확장으로 가능
